## Context

The node currently logs raw acceleration data at the DEBUG log level to standard output/files. However, to facilitate offline analysis of node behavior (like plotting dynamic behavior), we need to log tilt data alongside acceleration data, formatted in an easy-to-parse structure like CSV. Doing this directly on standard debug logs mixed with normal debug prints is inconvenient and clutters the data. We also do not want these large raw datasets sent to the remote server to conserve bandwidth and avoid cluttering server storage.

## Goals / Non-Goals

**Goals:**
- Provide a clear configuration to toggle CSV debug logging on/off.
- Create a dedicated file for CSV logging or format standard debug logs so that CSV parsing is seamless.
- Overwrite previous logs on each run to prevent storage from filling up quickly during repetitive manual testing.
- Ensure that logs generated by this debug mode are excluded from the MQTT/server transmission pipeline.

**Non-Goals:**
- Parsing CSV files directly on the ESP32.
- Implementing an automatic log rotation system based on file size for these specific debug logs (a simple overwrite on startup is sufficient).
- Uploading these CSV logs to the server.

## Decisions

- **File Output Strategy**: The logger component will manage the creation of a CSV debug file (e.g., `accel_tilt_debug.csv`). Upon initialization, if the CSV debug configuration is enabled, the logger will open the file with `w` mode (overwrite) rather than `a` mode (append), thus rewriting the file on startup.
- **Log Format**: The CSV will contain a header row: `timestamp_ms,accel_x,accel_y,accel_z,tilt_x,tilt_y,tilt_z` to standardize the offline parsing.
- **Server Exclusion**: The monitor and logger pipeline will either use a specific log level or a direct file write mechanism that bypasses the queue responsible for transmitting data to the server. Given existing structures, it might be simpler to write directly to the microSD card in the CSV format and bypass the warning/server log queues entirely.
- **Tilt Calculation**: The monitor component will be updated to output tilt data along with acceleration at the debug level.

## Risks / Trade-offs

- **Risk**: Frequent writes to the SD card for raw data (e.g., at 100Hz) could cause I/O bottlenecks.
  - **Mitigation**: Use buffering before writing to the SD card if necessary, or ensure the SDIO 1-bit interface is capable of handling the bandwidth. (Averaging 50-100 bytes per line at 100Hz is ~10KB/s, which SDIO can easily handle).
- **Risk**: Forgetting to disable the debug flag could degrade long-term deployment performance or SD card lifespan.
  - **Mitigation**: Clearly document this as a debug-only feature.
